\begin{abstract}
We present \textsc{GraphOS}, a dataflow microkernel that replaces traditional \texttt{switch}-based packet classification with ONNX graphs on an Intel NPU.
The core insight is that routing logic is \emph{isomorphic} to matrix multiplication (Theorem~\ref{thm:switch-matmul}): a hand-crafted weight matrix $\mat{W}\vect{x}$ replicates the switch table, and chaining it with a learned classifier via tensor adapters \emph{provably improves} routing decisions (Theorem~\ref{thm:composition})---\emph{todo es el grafo}.
A thread-per-node dataflow scheduler with bounded channels guarantees deadlock-free execution, and a C++20 native runtime eliminates Python overhead via lock-free SPSC channels, zero-copy \texttt{ov::Tensor} wrapping, and AVX2 SIMD normalization.
To address the $1{,}850\times$ single-packet NPU latency penalty, we introduce \emph{GPNPU}: a diamond-DAG pipeline where CPU fast-path rules classify ``obvious'' packets at ${\sim}0.38\,\mu\text{s}$ while only uncertain traffic reaches the NPU via triple-buffered async inference, achieving 80\% fast-path ratio with per-packet latency instrumentation (p50/p95/p99) confirming that fast-path p95 $\ll$ hard-path p95.
All results are validated on an Intel Meteor Lake NPU 3720 at 11~TOPS.
\end{abstract}

\section{Introduction and Motivation}
\label{sec:intro}
% ═══════════════════════════════════════════════════════════════════

Traditional network stacks classify packets using nested conditional branches---\texttt{if-else} chains over protocol numbers, port ranges, and header flags.
These sequential decision trees are fast on CPUs (${\sim}0.38\,\mu\text{s}$ per packet on modern hardware) but architecturally rigid: every new classification rule requires code changes, recompilation, and redeployment.

We propose replacing the entire classification pipeline with a \emph{learned} function expressed as a directed acyclic graph (DAG) of tensor operations, compiled and executed on a dedicated Neural Processing Unit (NPU).
The key insight is:

\begin{quote}
\emph{``Todo es el grafo''} --- Every decision, from packet-to-tensor encoding to multi-program routing composition, can be expressed as a node in a dataflow graph operating on tensors.
\end{quote}

\subsection{Contributions}

\begin{enumerate}[label=\textbf{C\arabic*.}]
  \item A formal tensor encoding $\phi$ that losslessly maps 64-byte packet headers to $\R^{64}$ while satisfying the input constraints of NPU accelerators (Section~\ref{sec:encoding}).
  \item A two-layer MLP $f_\theta: \R^{64} \to \R^3$ trained on 150{,}000 synthetic packets achieving 99.03\% classification accuracy with only 3{,}235 parameters (Section~\ref{sec:classifier}).
  \item A proof that traditional \texttt{switch}-based routing is isomorphic to a single matrix-vector product $\mat{W}\vect{x}$ with hand-crafted weights (Section~\ref{sec:route-table}).
  \item A program composition framework proving that chained ONNX programs with shape adapters produce \emph{better} routing decisions than independent programs (Section~\ref{sec:composition}).
  \item A formally verified dataflow scheduler using Kahn's topological sort with bounded-channel backpressure guaranteeing deadlock-free execution (Section~\ref{sec:dataflow}).
  \item Complete hardware characterization of the Intel NPU 3720 via OpenVINO, including latency measurements and batch-amortization analysis (Section~\ref{sec:hardware}).
  \item A hybrid GPNPU architecture with CPU fast-path rules and NPU hard-path async inference in a diamond-DAG pipeline, achieving ${\sim}80\%$ fast-path ratio and formally proven packet conservation (Section~\ref{sec:gpnpu}).
\end{enumerate}

\section{Hardware Platform and Execution Model}
\label{sec:hardware}
% ═══════════════════════════════════════════════════════════════════

The target hardware is an Intel Core Ultra 7 155H (Meteor Lake) with a dedicated NPU 3720 (11~TOPS INT8, systolic array). ONNX models are compiled once via OpenVINO 2026.0 with static shapes (Appendix~\ref{app:openvino}); at runtime, each inference transfers a batch to the NPU and reads back results.

\begin{definition}[NPU Inference Latency Model]
\label{def:latency}
For a batch of $B$ packets, the total inference latency is:
\begin{equation}
T_{\text{NPU}}(B) = T_{\text{setup}} + T_{\text{H}\to\text{D}}(B) + T_{\text{compute}}(B) + T_{\text{D}\to\text{H}}(B)
\end{equation}
The per-packet amortized cost is $\bar{T}_{\text{NPU}} = T_{\text{NPU}}(B) / B$.
\end{definition}

Empirically, $T_{\text{NPU}}(1) \approx 703\,\mu\text{s}$ while $T_{\text{CPU-switch}}(1) \approx 0.38\,\mu\text{s}$---an overhead factor of ${\sim}1{,}850\times$ for single packets. Batching amortizes $T_{\text{setup}}$, making the NPU competitive at $B \geq 64$.

\begin{property}[Static Shape Constraint]
\label{prop:static}
The Intel NPU 3720 requires \emph{static tensor shapes} at compilation time. The batch size $B$ must be fixed at ONNX export; partial batches are zero-padded to $B$.
\end{property}

\section{Tensor Encoding}
\label{sec:encoding}
% ═══════════════════════════════════════════════════════════════════

\begin{definition}[Packet Encoding Function]
\label{def:encoding}
Let $\mathcal{P} = \{0, 1, \ldots, 255\}^*$ be the set of all byte strings. The encoding function $\phi: \mathcal{P} \to [0,1]^D$ where $D = 64$ is defined as:
\begin{equation}
\phi(\vect{p})_i = \frac{p_i}{255}, \quad i \in \{0, 1, \ldots, D-1\}
\end{equation}
with truncation for $i \geq |\vect{p}|$ (zero-padding). The encoding is bounded ($\phi(\vect{p}) \in [0,1]^D$), injective on the $D$-byte prefix, and fixed-dimensional.
\end{definition}

The 64-byte tensor encodes Ethernet + IPv4 + Transport headers (Figure~\ref{fig:packet-layout}). The classification-critical bytes are \textbf{TTL} (offset~22), \textbf{Protocol} (offset~23), and \textbf{DstPort} (offset~36). After normalization: $\text{TCP} \mapsto 6/255 \approx 0.024$, $\text{UDP} \mapsto 17/255 \approx 0.067$, port~80 $\mapsto 80/255 \approx 0.314$. Batches of $B$ packets are encoded into $\tensor{X} \in [0,1]^{B \times D}$ in $\Theta(BD)$ time, zero-padding unused rows.

\section{Packet Classification: The MLP Classifier $f_\theta$}

\subsection{Architecture}

\begin{definition}[RouterGraph MLP]
\label{def:mlp}
The packet classifier $f_\theta: \R^{D} \to \R^{C}$ is a feedforward neural network with $D = 64$, $C = 3$, hidden dimension $H = 32$:
\begin{align}
\vect{h}_1 &= \ReLU(\mat{W}_1 \vect{x} + \vect{b}_1), & \mat{W}_1 \in \R^{H \times D},\; \vect{b}_1 \in \R^{H} \\
\vect{h}_2 &= \ReLU(\mat{W}_2 \vect{h}_1 + \vect{b}_2), & \mat{W}_2 \in \R^{H \times H},\; \vect{b}_2 \in \R^{H} \\
\vect{z} &= \mat{W}_3 \vect{h}_2 + \vect{b}_3, & \mat{W}_3 \in \R^{C \times H},\; \vect{b}_3 \in \R^{C}
\end{align}
where $\ReLU(a) = \max(0, a)$ applied element-wise, and $\vect{z} \in \R^C$ are raw logits (no softmax).
\end{definition}

\subsection{Parameter Count}

\begin{theorem}[Parameter Count]
The total number of trainable parameters is:
\begin{equation}
|\theta| = \underbrace{(D \cdot H + H)}_{\text{Layer 1}} + \underbrace{(H \cdot H + H)}_{\text{Layer 2}} + \underbrace{(H \cdot C + C)}_{\text{Layer 3}}
= (64 \cdot 32 + 32) + (32 \cdot 32 + 32) + (32 \cdot 3 + 3) = 3{,}235
\end{equation}
\end{theorem}

\subsection{Classification Decision Rule}

\begin{definition}[Argmax Classification]
The predicted class $\hat{y}$ for input $\vect{x}$ is:
\begin{equation}
\hat{y} = \argmax_{c \in \{0,1,2\}} f_\theta(\vect{x})_c
\end{equation}
where class labels are: HTTP $= 0$, DNS $= 1$, OTHER $= 2$.

\textbf{Note:} We deliberately omit the softmax layer at inference time. Since $\softmax$ is monotonic, $\argmax(\vect{z}) = \argmax(\softmax(\vect{z}))$, and skipping it saves compute on the NPU.
\end{definition}

\subsection{Training Procedure}

\begin{algorithm}[H]
\caption{\textsc{TrainClassifier}: SGD training with cross-entropy loss}
\label{alg:train}
\begin{algorithmic}[1]
\Require Dataset $\mathcal{D} = \{(\vect{x}_i, y_i)\}_{i=1}^{N}$ with $N = 150{,}000$, epochs $E = 20$, batch size $B_{\text{train}} = 256$, learning rate $\eta = 10^{-3}$
\Ensure Trained parameters $\theta^*$
\State Split $\mathcal{D}$ into $\mathcal{D}_{\text{train}}$ (90\%) and $\mathcal{D}_{\text{val}}$ (10\%)
\State Initialize $\theta$ randomly (PyTorch defaults)
\State $\textit{best\_acc} \gets 0$
\For{$e = 1$ \textbf{to} $E$}
  \For{each mini-batch $\mathcal{B} \subset \mathcal{D}_{\text{train}}$, $|\mathcal{B}| = B_{\text{train}}$}
    \State $\vect{z} \gets f_\theta(\mat{X}_{\mathcal{B}})$ \Comment{Forward pass: $(B_{\text{train}}, D) \to (B_{\text{train}}, C)$}
    \State $\mathcal{L} \gets \textsc{CrossEntropyLoss}(\vect{z}, \vect{y}_{\mathcal{B}})$ \Comment{Eq.~\eqref{eq:celoss}}
    \State $\theta \gets \theta - \eta \cdot \nabla_\theta \mathcal{L}$ \Comment{Adam optimizer step}
  \EndFor
  \State $\textit{val\_acc} \gets \textsc{Evaluate}(\theta, \mathcal{D}_{\text{val}})$
  \If{$\textit{val\_acc} > \textit{best\_acc}$}
    \State $\theta^* \gets \theta$; \quad $\textit{best\_acc} \gets \textit{val\_acc}$
  \EndIf
\EndFor
\State \Return $\theta^*$ \Comment{Best validation checkpoint}
\end{algorithmic}
\end{algorithm}

\begin{definition}[Cross-Entropy Loss]
\label{def:celoss}
\begin{equation}
\mathcal{L}(\vect{z}, y) = -\log\frac{\exp(z_y)}{\sum_{c=0}^{C-1} \exp(z_c)} = -z_y + \log \sum_{c=0}^{C-1} \exp(z_c)
\label{eq:celoss}
\end{equation}
where $\vect{z} = f_\theta(\vect{x}) \in \R^C$ and $y \in \{0, \ldots, C-1\}$ is the true class.
\end{definition}

\subsection{Synthetic Data Generation}

\begin{definition}[Packet Generation Rules]
\label{def:packet-gen}
Three generators $G_{\text{HTTP}}, G_{\text{DNS}}, G_{\text{OTHER}}: \Omega \to \{0,\ldots,255\}^{64}$ produce synthetic packets:
\begin{align}
G_{\text{HTTP}} &: \text{protocol} = 6\;\text{(TCP)}, \quad \text{dport} \in \{80, 443, 8080, 8443\} \\
G_{\text{DNS}} &: \text{protocol} = 17\;\text{(UDP)}, \quad \text{dport} = 53 \\
G_{\text{OTHER}} &: \text{protocol} \in \{1, 6, 17\}, \quad \text{dport} \notin \{53, 80, 443, 8080, 8443\}
\end{align}
All other fields (MACs, IPs, sequence numbers, payloads) are uniformly random.
The dataset contains $N/C = 50{,}000$ samples per class (perfectly balanced).
\end{definition}

\section{Route Table as Matrix Multiplication}
\label{sec:route-table}
% ═══════════════════════════════════════════════════════════════════

\subsection{The Isomorphism Theorem}

We now prove the central theoretical result: traditional \texttt{switch}-based routing is mathematically equivalent to a single matrix-vector product.

\begin{definition}[CPU Switch Classifier]
\label{def:switch}
The traditional routing function $g: \{0,\ldots,255\}^{64} \to \{0,1,2,3\}$ is:
\begin{equation}
g(\vect{p}) = \begin{cases}
\text{LOCAL} & \text{if } p_{23} = 17 \wedge p_{37} = 53 \\
\text{FORWARD} & \text{if } p_{23} = 6 \wedge p_{37} \in \{80, 443, \ldots\} \\
\text{DROP} & \text{if } p_{22} = 0 \\
\text{MONITOR} & \text{otherwise}
\end{cases}
\end{equation}
This requires sequential conditional evaluation: $\BigO(R)$ branches for $R$ routes.
\end{definition}

\begin{theorem}[Switch-to-Matmul Isomorphism]
\label{thm:switch-matmul}
For any finite set of routing rules based on byte-value thresholds, there exists a weight matrix $\mat{W} \in \R^{R \times D}$ such that:
\begin{equation}
g(\vect{p}) = \argmax_{r \in \{0,\ldots,R-1\}} (\mat{W} \cdot \phi(\vect{p}))_r
\end{equation}
where $\phi$ is the packet encoding from Definition~\ref{def:encoding}.
\end{theorem}

\begin{proof}
We construct $\mat{W}$ explicitly. Each route $r$ is characterized by a set of byte conditions. For condition ``byte at offset $\delta$ should have value $v$'', we set $W_{r,\delta}$ to a large positive weight $\alpha$ when the normalized value $v/255$ is high, or a large negative weight $-\beta$ to penalize undesired values.

The key insight is that after normalization, different protocol numbers map to different magnitudes: $\text{UDP}(17) \mapsto 0.067$ while $\text{TCP}(6) \mapsto 0.024$. By choosing appropriate weight magnitudes, we can create decision boundaries that separate these values.

Since $\argmax$ selects the route with the highest score, and each route's score is a weighted sum of byte features, the entire routing table is expressed as $\mat{W}\vect{x}$ where $\vect{x} = \phi(\vect{p})$.
\end{proof}

\subsection{Explicit Weight Construction}

\begin{definition}[Route Table Weight Matrix]
\label{def:route-weights}
The weight matrix $\mat{W} \in \R^{4 \times 64}$ for routing is constructed as:

\begin{equation}
\mat{W} = \begin{pmatrix}
\vect{w}_{\text{LOCAL}} \\
\vect{w}_{\text{FORWARD}} \\
\vect{w}_{\text{DROP}} \\
\vect{w}_{\text{MONITOR}}
\end{pmatrix}
\end{equation}

with non-zero entries:
\begin{align}
&\textbf{LOCAL}:      & W_{0,23} &= +100 \label{eq:local}\\
&\textbf{FORWARD}:    & W_{1,23} &= -30, \; W_{1,37} = +25, \; W_{1,36} = +8 \label{eq:forward}\\
&\textbf{DROP}:       & W_{2,22} &= -20, \; W_{2,j} \mathrel{+}= 0.015 \;\forall j \label{eq:drop}\\
&\textbf{MONITOR}:    & W_{3,23} &= -100, \; W_{3,22} = +12 \label{eq:monitor}
\end{align}
\end{definition}

\subsection{Computational Complexity Comparison}

\begin{theorem}[Matmul vs Switch Complexity]
\label{thm:complexity}
For $R$ routes and $D$-dimensional packet tensors:
\begin{align}
T_{\text{switch}}(B) &= \BigO(B \cdot R) & \text{(sequential per packet)} \\
T_{\text{matmul}}(B) &= \BigO(B \cdot R \cdot D) & \text{(CPU)} \\
T_{\text{NPU-matmul}}(B) &= \BigO(T_{\text{setup}} + B \cdot R \cdot D / P) & \text{(NPU, $P$ parallel MACs)}
\end{align}
where $P$ is the NPU's MAC parallelism. For NPU 3720 at 11~TOPS with INT8: $P \approx 11 \times 10^{12}$ ops/sec.
\end{theorem}

\begin{remark}
Although the matmul has higher asymptotic complexity than the switch ($\BigO(RD)$ vs $\BigO(R)$), the NPU executes all multiplications in parallel. At $B = 64$, $R = 4$, $D = 64$: the matmul requires $64 \times 4 \times 64 = 16{,}384$ MACs, which the NPU can execute in ${\sim}1.5\,\mu\text{s}$ (versus ${\sim}0.4\,\mu\text{s}$ per packet on CPU). The key advantage is \emph{programmability}: changing routing rules only requires updating $\mat{W}$, not recompiling code.
\end{remark}

\subsection{Motivation: Feature Fusion via Program Chaining}

The route table $\mat{W}\vect{x}$ operates solely on raw packet bytes. But the classifier $f_\theta$ has already computed a rich representation of the packet's traffic type. Can we \emph{compose} these programs so that downstream routing benefits from upstream classification?

\begin{definition}[Composed Router]
\label{def:composed}
The composed routing function $g^*: \R^D \to \R^R$ takes as input the concatenation of the raw packet tensor and the classifier's logits:
\begin{equation}
g^*(\vect{x}) = \mat{W}^* \cdot \concat(\vect{x},\; f_\theta(\vect{x}))
\end{equation}
where $\mat{W}^* \in \R^{R \times (D+C)}$ and $\concat: \R^D \times \R^C \to \R^{D+C}$ is vector concatenation.
\end{definition}

\subsection{Composed Weight Matrix}

\begin{definition}[Composed Router Weights]
\label{def:composed-weights}
The weight matrix $\mat{W}^* \in \R^{4 \times 67}$ has two regions:

\vspace{0.5em}
\noindent\textbf{Packet region} (columns 0--63): byte-level heuristics similar to the original route table.

\noindent\textbf{Logit region} (columns 64--66): classifier output signals.

\begin{equation}
\mat{W}^* = \left[\;\underbrace{\mat{W}_{\text{packet}}}_{\R^{4 \times 64}}\;\middle|\;\underbrace{\mat{W}_{\text{logit}}}_{\R^{4 \times 3}}\;\right]
\end{equation}

where the logit weights encode:
\begin{equation}
\mat{W}_{\text{logit}} = \begin{pmatrix}
0 & +15 & 0 \\
+15 & 0 & 0 \\
0 & 0 & +12 \\
-5 & -5 & -5
\end{pmatrix}
\quad
\begin{array}{l}
\leftarrow \text{LOCAL: boost DNS logit} \\
\leftarrow \text{FORWARD: boost HTTP logit} \\
\leftarrow \text{DROP: boost OTHER logit} \\
\leftarrow \text{MONITOR: penalize all}
\end{array}
\label{eq:logit-weights}
\end{equation}
\end{definition}

\subsection{Advantage of Composition}

\begin{theorem}[Composition Improves Routing]
\label{thm:composition}
Let $\vect{z} = f_\theta(\vect{x}) \in \R^3$ be the classifier logits for packet $\vect{x}$. For the composed router:
\begin{enumerate}[label=(\roman*)]
  \item When $z_{\text{DNS}} \gg z_{\text{HTTP}}, z_{\text{OTHER}}$ (strong DNS classification), the LOCAL score receives a boost of $+15 \cdot z_{\text{DNS}}$, independent of packet byte noise.
  \item When $z_{\text{HTTP}} \gg z_{\text{DNS}}, z_{\text{OTHER}}$ (strong HTTP classification), the FORWARD score receives $+15 \cdot z_{\text{HTTP}}$.
  \item When all $z_c$ are low (uncertain classification), the MONITOR score is \emph{least penalized} (since $-5 \cdot z_c \approx 0$ when $z_c \ll 0$), causing uncertain packets to be monitored.
\end{enumerate}
\end{theorem}

\begin{proof}
The composed score for route $r$ is:
\begin{equation}
s_r^* = \underbrace{\vect{w}_{r,\text{packet}}^\top \vect{x}}_{\text{byte heuristic}} + \underbrace{\vect{w}_{r,\text{logit}}^\top \vect{z}}_{\text{classifier signal}}
\end{equation}
For case (i): $s_{\text{LOCAL}}^* = \vect{w}_{0}^\top \vect{x} + 15 \cdot z_{\text{DNS}}$. Since $z_{\text{DNS}} > 0$ by hypothesis, this strictly increases the LOCAL score beyond the raw byte heuristic alone.

For case (iii): MONITOR receives $-5(z_0 + z_1 + z_2)$. When all $z_c < 0$ (uncertain), this becomes $+5|z_0 + z_1 + z_2| > 0$, boosting MONITOR. Other routes are boosted only by their respective positive logit, which is $< 0$ in the uncertain case.
\end{proof}

\subsection{Tensor Adapter Framework}

\begin{definition}[Tensor Adapter]
\label{def:adapter}
A tensor adapter $A = (\mathcal{S}, \tau)$ consists of:
\begin{itemize}
  \item An adapter specification $\mathcal{S} = (\text{name}, \{\text{input\_shapes}\}, \text{output\_shape})$
  \item A pure function $\tau: \prod_{k} \R^{s_k} \to \R^{s_{\text{out}}}$ mapping named input tensors to a single output tensor
\end{itemize}
\end{definition}

Two factory functions are provided:

\begin{align}
\concat_B^{L,R}(\vect{x}, \vect{y}) &: \R^{B \times L} \times \R^{B \times R} \to \R^{B \times (L+R)} \label{eq:concat} \\
\pad_B^{D \to D'}(\vect{x}) &: \R^{B \times D} \to \R^{B \times D'}, \quad D' \geq D \label{eq:pad}
\end{align}

where $\concat$ is implemented via \texttt{np.concatenate} along axis 1, and $\pad$ via \texttt{np.pad} with zero-fill.

\subsection{Pipeline Execution}

\begin{algorithm}[H]
\caption{\textsc{PipelineExecute}: Execute a composed program pipeline}
\label{alg:pipeline}
\begin{algorithmic}[1]
\Require Runtime $\mathcal{R}$, input tensor $\tensor{X}_0 \in \R^{B \times D}$, pipeline stages $S = [(k_i, n_i, o_i)]_{i=1}^{m}$
\Ensure Output tensor $\tensor{Y}$
\State $\tensor{X}_{\text{current}} \gets \tensor{X}_0$
\State $\tensor{X}_{\text{raw}} \gets \tensor{X}_0$ \Comment{Save for raw passthrough}
\For{$i = 1$ \textbf{to} $m$}
  \If{$k_i = \text{``program''}$}
    \State $\tensor{X}_{\text{current}} \gets \mathcal{R}.\textsc{Execute}(n_i, \tensor{X}_{\text{current}})$ \Comment{NPU inference}
  \ElsIf{$k_i = \text{``adapter''}$}
    \State Build input map from $o_i.\text{spec}$
    \If{adapter is concat-type}
      \State $\textit{inputs} \gets \{\text{``left''}: \tensor{X}_{\text{raw}},\; \text{``right''}: \tensor{X}_{\text{current}}\}$
    \Else
      \State $\textit{inputs} \gets \{\text{``in''}: \tensor{X}_{\text{current}}\}$
    \EndIf
    \State $\tensor{X}_{\text{current}} \gets o_i.\textsc{Execute}(\textit{inputs})$ \Comment{CPU tensor transform}
  \EndIf
\EndFor
\State \Return $\tensor{X}_{\text{current}}$
\end{algorithmic}
\end{algorithm}

\begin{theorem}[Pipeline Shape Compatibility]
\label{thm:pipeline-valid}
A pipeline with stages $S = [(k_i, n_i, o_i)]$ is shape-valid if and only if for all consecutive stages $i, i+1$:
\begin{equation}
\text{output\_shape}(S_i) = \text{input\_shape}(S_{i+1})
\end{equation}
The \textsc{Validate} function checks this in $\BigO(m)$ time for $m$ stages.
\end{theorem}

\section{Dataflow Framework}
\label{sec:dataflow}
% ═══════════════════════════════════════════════════════════════════

\subsection{Formal Model}

\begin{definition}[Dataflow Graph]
\label{def:dataflow}
A dataflow graph is a DAG $G = (V, E, \text{ports}, \text{cap})$ where:
\begin{itemize}
  \item $V$ is a finite set of processing nodes
  \item $E \subseteq V \times V$ is a set of directed edges (data channels)
  \item $\text{ports}: V \to 2^{\text{InputPort}} \times 2^{\text{OutputPort}}$ maps nodes to their typed ports
  \item $\text{cap}: E \to \N^+$ assigns bounded capacity to each channel
\end{itemize}
\end{definition}

\begin{definition}[Bounded FIFO Channel]
\label{def:channel}
A channel $\mathcal{C}_k$ with capacity $k \in \N^+$ is a FIFO queue with operations:
\begin{itemize}
  \item $\textsc{Put}(\mathcal{C}_k, x)$: Enqueue $x$. \textbf{Blocks} if $|\mathcal{C}_k| = k$ (backpressure).
  \item $\textsc{Get}(\mathcal{C}_k)$: Dequeue and return head. \textbf{Blocks} if $|\mathcal{C}_k| = 0$.
\end{itemize}
Both operations are $\BigO(1)$ amortized, implemented via \texttt{queue.Queue}.
\end{definition}

Before scheduling, edges are validated via DFS-based cycle detection in $\BigO(|V| + |E|)$ time (Appendix~\ref{app:dataflow-details}).

\subsection{Topological Sort: Kahn's Algorithm}

\begin{algorithm}[H]
\caption{\textsc{TopologicalSort}: Kahn's algorithm for dataflow scheduling order}
\label{alg:kahn}
\begin{algorithmic}[1]
\Require DAG $G = (V, E)$
\Ensure Topological ordering $\sigma: V \to \{1, \ldots, |V|\}$, or error if cycle detected
\State $\textit{in\_degree}[v] \gets |\{u : (u,v) \in E\}|$ for all $v \in V$ \Comment{Compute in-degrees}
\State $Q \gets \{v \in V : \textit{in\_degree}[v] = 0\}$ \Comment{Sources (no incoming edges)}
\State $\textit{order} \gets []$
\While{$Q \neq \emptyset$}
  \State $v \gets Q.\textsc{Dequeue}()$
  \State $\textit{order}.\textsc{Append}(v)$
  \For{each $(v, w) \in E$}
    \State $\textit{in\_degree}[w] \gets \textit{in\_degree}[w] - 1$
    \If{$\textit{in\_degree}[w] = 0$}
      \State $Q.\textsc{Enqueue}(w)$
    \EndIf
  \EndFor
\EndWhile
\If{$|\textit{order}| < |V|$}
  \State \textbf{raise} GraphError(``Cycle detected'')
\EndIf
\State \Return \textit{order}
\end{algorithmic}
\end{algorithm}

\begin{theorem}[Kahn's Algorithm Correctness]
Algorithm~\ref{alg:kahn} produces a valid topological ordering of $G$ if and only if $G$ is a DAG.
\end{theorem}
\begin{proof}
\textbf{Correctness}: Each dequeued node $v$ has $\textit{in\_degree}[v] = 0$, meaning all predecessors have already been placed in the ordering. This satisfies the topological constraint.

\textbf{Completeness}: If $G$ is a DAG, then at every step there exists at least one node with in-degree 0 (a source in the remaining subgraph). Thus the algorithm processes all $|V|$ nodes.

\textbf{Cycle detection}: If $|\textit{order}| < |V|$, then some nodes have in-degree $> 0$ even after removing all processed nodes, which implies a cycle in the remaining subgraph.
\end{proof}

\begin{theorem}[Kahn's Algorithm Complexity]
$\BigO(|V| + |E|)$ time, $\BigO(|V|)$ space.
\end{theorem}

\subsection{Thread-Per-Node Scheduler}

The scheduler topologically sorts $G$, launches one thread (Python \texttt{threading.Thread} / C++ \texttt{std::jthread}) per node, starts all threads concurrently, and joins them. Each node executes a setup$\to$process$\to$teardown lifecycle. Shutdown propagates via channel closure: when a source exhausts its input, it closes its output channels; downstream nodes receive \texttt{nullopt} / \texttt{\_STOP} and close their own outputs in turn. By induction on topological order, all nodes terminate in finite time (see Appendix~\ref{app:dataflow-details}).

\begin{invariant}[Deadlock Freedom]
\label{inv:deadlock}
A dataflow graph $G$ is deadlock-free if $G$ is a DAG (no cycles) and all channels have capacity $k \geq 1$.
\end{invariant}
\begin{proof}[Proof sketch]
Deadlock requires a circular wait. In a DAG, no cycles exist, so no circular dependency can form. Since $k \geq 1$, every producer can always make progress by putting one item before blocking. Total channel memory is bounded by $\sum_{(u,v) \in E} k_{(u,v)} \cdot s_{\max}$.
\end{proof}

\section{C++ Native Runtime}
\label{sec:cpp}
% ═══════════════════════════════════════════════════════════════════

The Python prototype (Phases 1--7) validated that NPU tensor operations can replace traditional packet classification. However, Python imposes fundamental performance ceilings: the GIL serializes threads, every packet traverses NumPy/ctypes copy boundaries, \texttt{queue.Queue} uses mutex locks, and the pybind11 OpenVINO wrapper adds per-call overhead. We perform a complete C++20 rewrite of the runtime to eliminate these bottlenecks while preserving the same ONNX models produced by the Python training pipeline.

\subsection{Architecture}

The C++ runtime eliminates Python's GIL via \texttt{std::jthread}-per-node with two key optimizations:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Lock-free SPSC channels}: A single-producer single-consumer ring buffer replaces \texttt{queue.Queue}. Atomic head/tail counters, \texttt{alignas(64)} cache-line padding, power-of-2 branchless modulo, and adaptive backoff (spin $\to$ \texttt{\_mm\_pause()} $\to$ yield) eliminate all mutex contention between nodes.

  \item \textbf{Zero-copy inference}: \texttt{ov::Tensor(element::f32, shape, ptr)} wraps the caller's \texttt{float*} directly---no \texttt{memcpy} on the hot path. AVX2 SIMD normalization converts 8 bytes $\to$ 8 floats per instruction ($8\times$ vs scalar). Batches use stack-allocated \texttt{std::array<OwnedPacket, 64>} to avoid heap allocation.
\end{enumerate}

\section{GPNPU: CPU Fast-Path / NPU Hard-Path Architecture}
\label{sec:gpnpu}
% ═══════════════════════════════════════════════════════════════════

The measurements in Section~\ref{sec:cpp} reveal a fundamental latency asymmetry: CPU rule evaluation completes in ${\sim}0.38\,\mu\text{s}$ while NPU single-inference costs ${\sim}700\,\mu\text{s}$. Sending \emph{all} packets through the NPU wastes accelerator bandwidth on traffic that can be classified deterministically from header bytes alone. The GPNPU architecture resolves this by splitting the packet stream into two paths:

\begin{itemize}[leftmargin=*]
  \item \textbf{Fast path (CPU)}: A constant-time rule classifier handles ``obvious'' packets---TTL=0 $\to$ DROP, UDP:53 $\to$ DNS/LOCAL, TCP:\{80, 443, 8080, 8443\} $\to$ HTTP/FORWARD---at sub-microsecond latency.
  \item \textbf{Hard path (NPU)}: Packets matching no rule are micro-batched with deadline-based flushing and dispatched to the NPU via triple-buffered asynchronous inference.
\end{itemize}

\noindent Both paths converge at a \textit{DecisionJoiner} before reaching the action dispatcher, forming a diamond-shaped DAG.

\subsection{Decision Types}

We introduce a unified output type for both paths:

\begin{definition}[Decision]
A \emph{Decision} is a tuple $(p, c, r, \pi, \gamma, s, \tau)$ where $p \in \{0,\ldots,255\}^{64}$ is the raw packet, $c \in \{0,1,2,-1\}$ is the class ID, $r \in \{0,1,2,3,-1\}$ is the route ID, $\pi \in \{\text{FAST}, \text{HARD}\}$ is the path kind, $\gamma \in [0,1]$ is the confidence ($1.0$ for rules, $\max_c f_\theta(x)_c$ for NPU), $s \in \mathbb{N}$ is a monotonic sequence number, and $\tau \in \mathbb{N}$ is the ingress timestamp (nanoseconds from \texttt{steady\_clock}) stamped at the FastPathClassifier.
\end{definition}

\begin{definition}[HardPathItem]
A \emph{HardPathItem} is a triple $(p, s, \tau)$: the raw packet, its sequence number, and the ingress timestamp. The timestamp $\tau$ is propagated through the batcher and NpuExecutorAsync so that end-to-end latency can be measured at the ActionDispatcher.
\end{definition}

\begin{definition}[NpuBatchItem]
An \emph{NpuBatchItem} is a fixed-capacity container of up to $B$ HardPathItems, with a \texttt{count} field recording the actual number of valid items and a \texttt{padded\_to} field always equal to $B$ (the ONNX static batch size).
\end{definition}

\begin{theorem}[Diamond DAG Validity]
\label{thm:diamond-dag}
The GPNPU topology $G = (\{S, F, B, N, J, D\}, E)$ with edges $E = \{(S,F), (F,J), (F,B), (B,N), (N,J), (J,D)\}$ is a valid DAG. Kahn's topological sort yields the ordering $[S, F, B, N, J, D]$ with in-degree sequence $[0, 1, 1, 1, 2, 1]$.
\end{theorem}

\begin{proof}
No path exists from any node back to itself. The only node with in-degree $> 1$ is $J$ (the joiner), receiving edges from $F$ and $N$, which lie on distinct simple paths from $F$. Since all edges point strictly rightward in the topological order, no cycle can exist.
\end{proof}

\subsection{FastPathClassifier: Rule-Based CPU Classification}

The classifier applies $k$ rules in priority order. Each rule is a constant-time predicate on raw packet bytes:

\begin{algorithm}[H]
\caption{\textsc{FastPathClassify}$(p)$: CPU fast-path decision}
\label{alg:fast-path}
\begin{algorithmic}[1]
\Require Packet $p \in \{0,\ldots,255\}^{64}$, sequence counter $s$
\Ensure Decision $d$ or HardPathItem $h$
\State $s \gets s + 1$
\If{$p[\text{OFFSET\_TTL}] = 0$}
  \State \Return $\text{Decision}(p, \text{OTHER}, \text{DROP}, \text{FAST}, 1.0, s)$
\EndIf
\State $\textit{proto} \gets p[\text{OFFSET\_PROTOCOL}]$
\State $\textit{port} \gets p[\text{OFFSET\_DST\_PORT}] \ll 8 \mathbin{|} p[\text{OFFSET\_DST\_PORT}+1]$
\If{$\textit{proto} = \text{UDP} \land \textit{port} = 53$}
  \State \Return $\text{Decision}(p, \text{DNS}, \text{LOCAL}, \text{FAST}, 1.0, s)$
\EndIf
\If{$\textit{proto} = \text{TCP} \land \textit{port} \in \{80, 443, 8080, 8443\}$}
  \State \Return $\text{Decision}(p, \text{HTTP}, \text{FORWARD}, \text{FAST}, 1.0, s)$
\EndIf
\State \Return $\text{HardPathItem}(p, s)$ \Comment{No rule matched $\to$ NPU}
\end{algorithmic}
\end{algorithm}

The port check uses a \texttt{constexpr} function (four equality comparisons) rather than a hash set, keeping the entire fast path in L1 instruction cache. The classifier emits to two output ports: \texttt{fast\_out} (typed \texttt{Decision}) and \texttt{hard\_out} (typed \texttt{HardPathItem}).

\begin{lemma}[Fast-Path Time Complexity]
\label{lem:fast-path-time}
Algorithm~\ref{alg:fast-path} runs in $\Theta(1)$ time per packet: at most 4 comparisons, 1 byte-shift, and 1 OR operation, all on stack-allocated data.
\end{lemma}

\subsection{HardBatcher: Deadline-Based Micro-Batching}

Packets that fail fast-path rules must be batched for NPU inference. The batcher accumulates HardPathItems and flushes under two conditions:

\begin{algorithm}[H]
\caption{\textsc{HardBatch}: Deadline-based micro-batching}
\label{alg:hard-batcher}
\begin{algorithmic}[1]
\Require Input channel $\mathcal{C}_{\text{in}}$, batch size $B$, min fill $m$, deadline $\delta$
\Ensure Padded NpuBatchItems on output channel $\mathcal{C}_{\text{out}}$
\State $\textit{batch} \gets \text{empty}$, $t_0 \gets \text{now}()$
\Loop
  \State $\textit{item} \gets \mathcal{C}_{\text{in}}.\text{try\_pop}()$
  \If{$\textit{item} \neq \bot$}
    \State $\textit{batch}.\text{push}(\textit{item})$
    \If{$|\textit{batch}| \geq B$} \Comment{Batch full}
      \State $\text{flush}(\textit{batch}, B)$; $\textit{batch} \gets \text{empty}$; $t_0 \gets \text{now}()$
    \EndIf
  \ElsIf{$\mathcal{C}_{\text{in}}.\text{closed}$}
    \If{$|\textit{batch}| > 0$} $\text{flush}(\textit{batch}, B)$ \EndIf
    \State $\mathcal{C}_{\text{out}}.\text{close}()$; \Return
  \ElsIf{$|\textit{batch}| \geq m \land (\text{now}() - t_0) \geq \delta$}
    \State $\text{flush}(\textit{batch}, B)$; $\textit{batch} \gets \text{empty}$; $t_0 \gets \text{now}()$
    \Comment{Deadline flush}
  \Else
    \State \text{adaptive\_backoff}() \Comment{spin $\to$ pause $\to$ yield}
  \EndIf
\EndLoop
\end{algorithmic}
\end{algorithm}

\begin{definition}[Flush Operation]
$\text{flush}(\textit{batch}, B)$ pads the batch to exactly $B$ items (the ONNX static shape) by zeroing the remaining slots, sets $\textit{padded\_to} = B$, and pushes the NpuBatchItem to $\mathcal{C}_{\text{out}}$.
\end{definition}

The parameters $m$ (default 12) and $\delta$ (default $150\,\mu\text{s}$) bound the latency--throughput tradeoff: setting $m = 1, \delta = 0$ minimizes latency (flush immediately), while $m = B$ maximizes batching efficiency.

\subsection{NpuExecutorAsync: Triple-Buffered Asynchronous Inference}

The NPU executor uses $K$ pre-allocated inference slots (default $K = 3$) for overlapped execution. Each slot contains an input buffer, output buffer, \texttt{ov::InferRequest}, and the batch context. The pimpl pattern hides all OpenVINO headers from the public API.

\begin{algorithm}[H]
\caption{\textsc{NpuExecutorAsync}: $K$-slot async inference}
\label{alg:npu-async}
\begin{algorithmic}[1]
\Require $K$ pre-allocated slots $\{(\textit{in}_k, \textit{out}_k, \textit{req}_k, \textit{ctx}_k)\}_{k=0}^{K-1}$
\Require Input channel $\mathcal{C}_{\text{in}}$ of NpuBatchItems
\Ensure Individual Decisions on $\mathcal{C}_{\text{out}}$
\State $i_s \gets 0$, $i_c \gets 0$, $\textit{inflight} \gets 0$
\Loop
  \If{$\textit{inflight} \geq K$}
    \State $\textsc{Collect}(i_c \bmod K)$; $i_c \gets i_c + 1$; $\textit{inflight} \gets \textit{inflight} - 1$
  \EndIf
  \State $\textit{batch} \gets \mathcal{C}_{\text{in}}.\text{pop}()$
  \If{$\textit{batch} = \bot$} \textbf{break} \EndIf
  \State $\textsc{Submit}(i_s \bmod K, \textit{batch})$; $i_s \gets i_s + 1$; $\textit{inflight} \gets \textit{inflight} + 1$
\EndLoop
\While{$\textit{inflight} > 0$} \Comment{Drain pending}
  \State $\textsc{Collect}(i_c \bmod K)$; $i_c \gets i_c + 1$; $\textit{inflight} \gets \textit{inflight} - 1$
\EndWhile
\State $\mathcal{C}_{\text{out}}.\text{close}()$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[H]
\caption{$\textsc{Submit}(k, \textit{batch})$ and $\textsc{Collect}(k)$}
\begin{algorithmic}[1]
\Procedure{Submit}{$k$, batch}
  \For{$i = 0$ \textbf{to} $|\textit{batch}| - 1$}
    \State $\textit{in}_k[i \cdot D : (i+1) \cdot D] \gets \phi(\textit{batch}[i].\textit{packet})$ \Comment{Normalize in-slot}
  \EndFor
  \For{$i = |\textit{batch}|$ \textbf{to} $B - 1$}
    \State $\textit{in}_k[i \cdot D : (i+1) \cdot D] \gets \vect{0}$ \Comment{Zero-pad}
  \EndFor
  \State $\textit{ctx}_k \gets \textit{batch}$; $\textit{req}_k.\text{start\_async}()$
\EndProcedure
\Procedure{Collect}{$k$}
  \State $\textit{req}_k.\text{wait}()$
  \For{$i = 0$ \textbf{to} $|\textit{ctx}_k| - 1$}
    \State $c \gets \argmax_{j} \textit{out}_k[i \cdot C + j]$
    \State $\gamma \gets \max_{j} \textit{out}_k[i \cdot C + j]$
    \State emit Decision$(\textit{ctx}_k[i].\textit{packet}, c, -1, \text{HARD}, \gamma, \textit{ctx}_k[i].s)$
  \EndFor
\EndProcedure
\end{algorithmic}
\end{algorithm}

\begin{theorem}[Triple-Buffer Overlap]
\label{thm:triple-buffer}
With $K = 3$ slots, the host can submit batch $n+1$ and collect batch $n-1$ while batch $n$ executes on the NPU. The steady-state throughput is:
\begin{equation}
\text{throughput}_{\text{async}} = \frac{B}{\max(T_{\text{NPU}}, T_{\text{host}})} \approx \frac{B}{T_{\text{NPU}}}
\end{equation}
where $T_{\text{NPU}} \gg T_{\text{host}}$ in practice (${586}\,\mu\text{s}$ vs ${<}50\,\mu\text{s}$ for normalization + submission).
\end{theorem}

\subsection{DecisionJoiner: Diamond Reconvergence}

The joiner polls both input ports (fast and hard) with non-blocking \texttt{try\_get()} and forwards any received Decision immediately. An optional \emph{ordered} mode uses a reorder buffer keyed by sequence number:

\begin{algorithm}[H]
\caption{\textsc{DecisionJoiner}: merge fast and hard paths}
\label{alg:joiner}
\begin{algorithmic}[1]
\Require InputPorts $\mathcal{C}_f$ (fast), $\mathcal{C}_h$ (hard); OutputPort $\mathcal{C}_o$; ordered flag
\State $\textit{f\_closed} \gets \text{false}$, $\textit{h\_closed} \gets \text{false}$, $\textit{next\_seq} \gets 0$
\While{$\lnot(\textit{f\_closed} \land \textit{h\_closed})$}
  \State $\textit{got} \gets \text{false}$
  \If{$\lnot\textit{f\_closed}$}
    \State $d \gets \mathcal{C}_f.\text{try\_get}()$
    \If{$d \neq \bot$} $\textsc{Emit}(d)$; $\textit{got} \gets \text{true}$
    \ElsIf{$\mathcal{C}_f.\text{closed\_and\_empty}$} $\textit{f\_closed} \gets \text{true}$
    \EndIf
  \EndIf
  \If{$\lnot\textit{h\_closed}$}
    \State $d \gets \mathcal{C}_h.\text{try\_get}()$
    \If{$d \neq \bot$} $\textsc{Emit}(d)$; $\textit{got} \gets \text{true}$
    \ElsIf{$\mathcal{C}_h.\text{closed\_and\_empty}$} $\textit{h\_closed} \gets \text{true}$
    \EndIf
  \EndIf
  \If{$\lnot\textit{got}$} $\text{adaptive\_backoff}()$ \EndIf
\EndWhile
\State Flush reorder buffer; $\mathcal{C}_o.\text{close}()$
\end{algorithmic}
\end{algorithm}

In unordered mode (default), $\textsc{Emit}(d)$ directly pushes $d$ to $\mathcal{C}_o$. In ordered mode, $\textsc{Emit}(d)$ inserts $d$ into a sorted map keyed by $d.s$, then flushes the contiguous prefix $\{s_{\text{next}}, s_{\text{next}}+1, \ldots\}$.

\begin{theorem}[Ordered Joiner Correctness]
\label{thm:ordered-joiner}
In ordered mode, if the FastPathClassifier assigns strictly monotonic sequence numbers to all packets, then the output sequence of the DecisionJoiner is a permutation-free reproduction of the input order. The reorder buffer size is bounded by $\BigO(B \cdot K)$ in the worst case, where $K$ is the number of inflight NPU batches.
\end{theorem}

\begin{proof}
Let $n$ packets enter the FastPathClassifier with sequences $0, 1, \ldots, n-1$. Fast-path packets (sequence $s$) arrive at the joiner immediately. Hard-path packets are delayed by at most $K$ batch intervals. The reorder buffer holds decisions with $s > s_{\text{next}}$ until the gap fills. Since every sequence number is emitted exactly once and the total count $n_f + n_h = n$, no packet is lost or duplicated.
\end{proof}

\subsection{ActionDispatcher: Unified Terminal Node}

The ActionDispatcher replaces the existing RouterSink for GPNPU pipelines. It accepts Decision items and dispatches to actions indexed by both \texttt{class\_id} and \texttt{route\_id}:

\begin{enumerate}[leftmargin=*]
  \item For each decision $d$, if $d.c \geq 0$, invoke all actions in $\texttt{class\_actions}[d.c]$.
  \item If $d.r \geq 0$, invoke all actions in $\texttt{route\_actions}[d.r]$.
  \item If neither fires, invoke the default action (if set).
  \item Increment \texttt{fast\_path\_count} or \texttt{hard\_path\_count} based on $d.\pi$.
  \item If $d.\tau > 0$, compute egress latency $\ell = (\tau_{\text{now}} - d.\tau) / 1000$ (in $\mu$s) and record it in the overall, fast-path, and hard-path \texttt{LatencyHistogram}s.
\end{enumerate}

Atomic counters (\texttt{std::atomic<uint64\_t>}) track path statistics without locking. Three \texttt{LatencyHistogram} instances collect per-packet end-to-end latency samples (overall, fast-path only, hard-path only) for percentile reporting.

\subsection{Latency Instrumentation: Percentile Metrics}
\label{sec:gpnpu-latency}

To quantify the fast-path advantage and identify hard-path bottlenecks, we instrument the GPNPU pipeline with per-packet end-to-end latency tracking.

\subsubsection{Ingress Timestamp Propagation}

A single \texttt{steady\_clock::now()} call per packet at the FastPathClassifier stamps the ingress time $\tau$ on both fast-path Decisions and hard-path HardPathItems. The timestamp propagates through the pipeline:

\begin{equation}
\text{FastPathClassifier} \xrightarrow{\tau} \begin{cases}
  \text{Decision}(d.\tau = \tau) & \text{fast path} \\
  \text{HardPathItem}(\tau) \to \text{HardBatcher} \to \text{NpuExecutorAsync} \xrightarrow{d.\tau = \tau} \text{Decision} & \text{hard path}
\end{cases}
\end{equation}

The NpuBatchItem preserves $\tau$ through the items array (which is moved intact), and the NpuExecutorAsync copies $\tau$ from each HardPathItem to the emitted Decision in \texttt{collect\_oldest()}.

\subsubsection{LatencyHistogram}

\begin{definition}[LatencyHistogram]
A \emph{LatencyHistogram} is a dynamically-sized sample collector supporting $\BigO(n)$ percentile computation via \texttt{std::nth\_element}. Given $n$ recorded samples $\{x_1, \ldots, x_n\}$:
\begin{itemize}[leftmargin=*]
  \item $\operatorname{percentile}(p) = x_{(\lfloor p \cdot (n-1) / 100 \rfloor)}$ where $x_{(k)}$ is the $k$-th order statistic.
  \item \texttt{nth\_element} partitions the copy in $\BigO(n)$ average time (no full sort).
  \item Percentile calls are \emph{non-destructive}: each call copies the sample vector internally.
  \item The \texttt{reserve} parameter avoids reallocations during the measurement phase.
\end{itemize}
\end{definition}

\subsubsection{Egress Measurement and Reporting}

At the ActionDispatcher, each incoming Decision with $\tau > 0$ triggers an egress timestamp and latency computation:
\begin{equation}
\ell_i = \frac{\tau_{\text{now}} - d_i.\tau}{1000} \quad (\mu\text{s})
\end{equation}
The sample $\ell_i$ is recorded in three histograms: \texttt{overall}, and either \texttt{fast} or \texttt{hard} depending on $d_i.\pi$. The \texttt{run\_gpnpu} command reports:

\subsubsection{Warmup Phase}

Cold-start skew from NPU JIT compilation and cache population can distort latency percentiles. The \texttt{-{}-warmup $w$} flag (default $w = 100$) runs $w$ packets through a separate pipeline instance before the measurement phase. Warmup results are discarded entirely. This ensures that p95/p99 reflect steady-state behavior rather than initialization artifacts.

\begin{algorithm}[H]
\caption{GPNPU Latency Measurement with Warmup}
\begin{algorithmic}[1]
\Require Packets $P = \{p_1, \ldots, p_{w+n}\}$, warmup count $w$
\Ensure Latency percentiles for $\{p_{w+1}, \ldots, p_{w+n}\}$
\State $P_{\text{warm}} \gets P[1 \ldots w]$
\State $P_{\text{measure}} \gets P[w{+}1 \ldots w{+}n]$
\State \textsc{RunPipeline}($P_{\text{warm}}$) \Comment{Discard results}
\State $\text{pipeline} \gets \textsc{BuildGpnpuPipeline}(P_{\text{measure}})$
\State $\textsc{RunPipeline}(\text{pipeline})$
\State \Return $\text{pipeline.dispatcher.overall\_latency}.\{p50, p95, p99\}$
\end{algorithmic}
\end{algorithm}

\subsubsection{Expected Latency Distribution}

For the diamond pipeline with fast-path ratio $\alpha$, the expected latency distribution is bimodal:
\begin{itemize}[leftmargin=*]
  \item \textbf{Fast-path mode}: $\ell_f \approx T_{\text{channel}} + T_{\text{joiner}} + T_{\text{dispatch}} = \BigO(1)\,\mu\text{s}$. These packets traverse only lock-free SPSC channels and CPU logic.
  \item \textbf{Hard-path mode}: $\ell_h \approx T_{\text{batch\_fill}} + T_{\text{NPU}} + T_{\text{collect}} + T_{\text{joiner}} + T_{\text{dispatch}} \approx \delta + T_{\text{NPU}}$. At $\delta = 150\,\mu\text{s}$ and $T_{\text{NPU}} = 586\,\mu\text{s}$, the hard-path p50 is expected near $700\,\mu\text{s}$.
\end{itemize}

\noindent The overall p95 is dominated by the hard-path tail, confirming the architectural hypothesis that fast-path p95 $\ll$ hard-path p95. This validates the GPNPU split: packets that \emph{can} be classified by rules should \emph{never} pay NPU latency.

\subsection{Formal Properties}

\begin{invariant}[Packet Conservation]
\label{inv:gpnpu-conservation}
For $n$ packets entering the FastPathClassifier:
\begin{equation}
n_f + n_h = n \quad \text{and} \quad n_{\text{dispatcher}} = n_f + n_h = n
\end{equation}
where $n_f$ is the fast-path count, $n_h$ is the hard-path count, and $n_{\text{dispatcher}}$ is the total decisions received by the ActionDispatcher. No packet is created, duplicated, or lost in the diamond DAG.
\end{invariant}

\begin{proof}
The FastPathClassifier emits exactly one output per input (either to \texttt{fast\_out} or \texttt{hard\_out}). The HardBatcher emits all received items (flushing on close). The NpuExecutorAsync emits one Decision per HardPathItem in each batch (ignoring padding). The DecisionJoiner forwards all items from both inputs. Therefore $n_{\text{dispatcher}} = n_f + n_h = n$.
\end{proof}

\begin{invariant}[Deadlock Freedom]
The GPNPU diamond DAG satisfies Invariant~\ref{inv:deadlock} (deadlock-free scheduling). The DecisionJoiner uses non-blocking \texttt{try\_get()} on both inputs, preventing deadlock from asymmetric path latencies. Channel capacities are sized at $4B$ on the fast path and $K+1$ on the NPU path to prevent backpressure blocking.
\end{invariant}

\begin{invariant}[NPU Shape Compliance]
Every NpuBatchItem has $\texttt{padded\_to} = B$, satisfying the ONNX static shape constraint. The HardBatcher zero-pads partial batches and the NpuExecutorAsync normalizes within its pre-allocated slot buffers. Only the first $\texttt{count}$ results are emitted; padded outputs are discarded.
\end{invariant}

\subsection{Fast-Path Ratio Analysis}

For typical enterprise traffic, the fast-path rules cover:
\begin{itemize}
  \item HTTP/HTTPS traffic (ports 80, 443) --- typically 50--70\% of traffic volume
  \item DNS queries (port 53) --- 5--10\% of traffic
  \item TTL-expired packets --- $< 1\%$
\end{itemize}

With the synthetic 5-pattern test traffic (TTL=0, DNS, HTTP:80, HTTPS:443, random), the measured fast-path ratio is exactly $4/5 = 80\%$, leaving only 20\% of packets for NPU processing.

\begin{corollary}[Effective Throughput]
\label{cor:effective-throughput}
Let $\alpha \in [0,1]$ be the fast-path ratio. The effective per-packet latency is:
\begin{equation}
T_{\text{eff}} = \alpha \cdot T_{\text{CPU}} + (1 - \alpha) \cdot \frac{T_{\text{NPU}}}{B}
\end{equation}
At $\alpha = 0.8$, $T_{\text{CPU}} = 0.38\,\mu\text{s}$, $T_{\text{NPU}} = 586\,\mu\text{s}$, $B = 64$:
\begin{equation}
T_{\text{eff}} = 0.8 \times 0.38 + 0.2 \times \frac{586}{64} \approx 0.30 + 1.83 = 2.13\,\mu\text{s}/\text{pkt}
\end{equation}
This represents a $\mathbf{4.3\times}$ improvement over sending all packets through NPU ($9.2\,\mu\text{s}/\text{pkt}$).
\end{corollary}

The GPNPU pipeline is validated by 39 unit and integration tests (95 total C++) covering all nodes, packet conservation ($n_f + n_h = n$), latency propagation, and percentile computation.

\subsection{The Synthetic--Real Domain Gap}

The recall gap is explained by the \emph{domain shift} between synthetic training data and real traffic:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Encrypted payloads}: The MLP observes all 64 input bytes. Real HTTPS traffic has TLS-encrypted payloads that are indistinguishable from random noise, matching the \textsc{OTHER} class distribution in synthetic training data.

  \item \textbf{Header-only signal}: The discriminative features (protocol byte at offset 23, destination port at offsets 36--37) occupy only 3 of 64 bytes (4.7\% of the input). The model's 32-neuron hidden layer had not learned to ignore the irrelevant encrypted payload bytes.

  \item \textbf{High precision despite low recall}: The model learned a \emph{conservative} decision boundary---it only predicts HTTP when confident, avoiding false positives.
\end{enumerate}

\subsection{Hybrid Retraining with Captured Traffic}
\label{sec:retrain}

To close the domain gap, we introduce a \texttt{HybridDataset} that combines synthetic and real packets:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Real traffic capture}: $3{,}000$ packets captured from the host's Wi-Fi interface via Npcap under normal browsing activity ($2{,}824$ HTTP, $28$ DNS, $148$ OTHER).
  \item \textbf{Header-based labeling}: A \texttt{label\_raw\_packet()} function assigns ground truth from protocol and port bytes, applying the same rules as the synthetic generators.
  \item \textbf{Oversampling}: Real packets are replicated $10\times$ to balance against the synthetic majority ($150{,}000$ synthetic vs.\ $35{,}000$ real = $185{,}000$ total).
  \item \textbf{Retraining}: 30 epochs, same architecture ($3{,}235$ parameters), Adam with $\eta = 10^{-3}$.
\end{enumerate}

\noindent The hybrid training exposes the model to real TLS-encrypted payload distributions paired with correct labels, teaching it to rely on header bytes for classification.

\section{Conclusion}
\label{sec:conclusion}
% ═══════════════════════════════════════════════════════════════════

The central thesis---\emph{todo es el grafo}---is validated through four results:

\begin{enumerate}
  \item \textbf{Switch$\leftrightarrow$Matmul Isomorphism} (Theorem~\ref{thm:switch-matmul}): Routing logic is a linear layer $\mat{W}\vect{x}$, executable on NPU without code changes. Chaining it with a learned classifier provably improves routing decisions (Theorem~\ref{thm:composition}).

  \item \textbf{C++ Native Runtime} (Section~\ref{sec:cpp}): Lock-free SPSC channels, zero-copy inference, and AVX2 normalization yield $109\text{K}$\,pkt/s (NPU) and $2.8\text{M}$\,pkt/s (CPU) sustained at $B{=}64$.

  \item \textbf{GPNPU Hybrid Architecture} (Section~\ref{sec:gpnpu}): A diamond-DAG splits traffic into CPU fast-path (${\sim}0.38\,\mu\text{s}$/pkt) and NPU hard-path (triple-buffered async), achieving $4.3\times$ speedup at 80\% fast-path ratio (Corollary~\ref{cor:effective-throughput}). Per-packet latency instrumentation (p50/p95/p99) confirms the bimodal distribution: fast-path p95 $\ll$ hard-path p95.

  \item \textbf{Live Traffic Validation and Hybrid Retraining} (Section~\ref{sec:live-eval}): End-to-end Npcap capture on live Wi-Fi traffic reveals a synthetic-to-real domain gap (47.8\% agreement) caused by TLS-encrypted payloads. Hybrid retraining on $3{,}000$ captured packets closes the gap completely: $\mathbf{100\%}$ agreement on 500 real packets (Table~\ref{tab:retrain-comparison}), $10{,}041$\,pkt/s sustained throughput on $5{,}000$ fresh packets---with zero code changes, only new ONNX weights.
\end{enumerate}

\noindent The CPU handles deterministic rule evaluation, the NPU handles learned classification, and the dataflow graph orchestrates both in a formally verified pipeline. The retraining result validates the central design principle: classification behavior is defined by tensor weights, not by code. Updating the model requires only re-exporting the ONNX file---the entire C++ runtime, dataflow graph, and GPNPU pipeline remain unchanged. The architecture is ready for DPDK kernel-bypass capture to close the remaining gap to line-rate processing on commodity NPU hardware.

\section{Dataflow Scheduler Details}
\label{app:dataflow-details}

\begin{algorithm}[H]
\caption{\textsc{WouldCreateCycle}: DFS-based cycle detection before adding edge}
\label{alg:cycle}
\begin{algorithmic}[1]
\Require Graph $G = (V, E)$, proposed edge $(u, v)$
\Ensure \textbf{true} if adding $(u, v)$ creates a cycle
\State $\textit{visited} \gets \emptyset$; $\textit{stack} \gets [v]$
\While{$\textit{stack} \neq \emptyset$}
  \State $w \gets \textit{stack}.\textsc{Pop}()$
  \If{$w = u$} \Return \textbf{true} \EndIf
  \If{$w \notin \textit{visited}$}
    \State $\textit{visited} \gets \textit{visited} \cup \{w\}$
    \ForAll{$(w, w') \in E$} $\textit{stack}.\textsc{Push}(w')$ \EndFor
  \EndIf
\EndWhile
\State \Return \textbf{false}
\end{algorithmic}
\end{algorithm}

Correctness: a cycle exists in $G \cup \{(u,v)\}$ iff there is a path $v \leadsto u$ in $G$. The DFS explores all nodes reachable from $v$. Complexity: $\BigO(|V| + |E|)$ time, $\BigO(|V|)$ space.

\begin{algorithm}[H]
\caption{\textsc{RunNode}: Single node lifecycle}
\label{alg:runnode}
\begin{algorithmic}[1]
\Require Node $v$ with inputs, outputs, and process loop
\State $v.\textsc{Setup}()$ \Comment{Initialize resources}
\Try
  \State $v.\textsc{Process}()$ \Comment{Main loop}
\Catch{Exception $e$}
  \State Close all output channels \Comment{Propagate shutdown}
\EndTry
\State $v.\textsc{Teardown}()$
\end{algorithmic}
\end{algorithm}

\textbf{Shutdown termination}: By induction on topological order. Sources terminate when exhausted and close outputs. Node at position $k > 0$: all predecessors at position $< k$ have terminated (induction); $v$ receives channel-closed and terminates.

\textbf{Bounded memory}: Each channel holds at most $k$ items of size $\leq s_{\max}$; total $M \leq \sum k_{(u,v)} \cdot s_{\max}$.

\begin{algorithm}[H]
\caption{OpenVINO Model Compilation}
\label{alg:openvino}
\begin{algorithmic}[1]
\Require ONNX model path $p$, target device $d \in \{\text{NPU}, \text{CPU}, \text{GPU}\}$
\Ensure Compiled inference request $r$

\end{algorithmic}
\end{algorithm}
